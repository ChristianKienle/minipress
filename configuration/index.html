# Configuration

You can configure certain aspects of miniPress by creating a configuration file at .minipress/config.js.

# Basics

A miniPress configuration file is a normal CommonJS-module and basically looks like this:

module.exports = {
  /* put your settings here */
}

# cwd

  • Type: string
  • Default: process.cwd()

# pages

  • Type: string
  • Default: $cwd/pages

# layouts

  • Type: Layouts
  • Default: $cwd/layouts

Specify the available layouts. The value of layouts can be

  • A Directory: (type string): In that case every *.vue-file in that directory automatically becomes a layout. The name of the layout will be the name of the corresponding file (post.vue will be available as the layout named post).
  • An Object: mapping the name of a layout to a *.vue-file. This gives you more control.
  • undefined.
type Layouts =
  | string /* path to a layouts-directory */
  | { [layoutName: string]: string
  ;

# components

  • Type: string
  • Default: $cwd/components

Specify a directory which contains *.vue-file. Components in that directory will be globally available â€“ for example in your markdown files.

# port

  • Type: number
  • Default: 4000

Specify the port used when something has to be served via HTTP. Usually you can override this via the CLI by using --port <port> â€“ for example:

minipress dev --port 8080

will run miniPress on port 8080 â€“ even though it may be set to something different in the configuration file.

# host

  • Type: string
  • Default: "0.0.0.0"

Specify the host to be used when something has to be served via HTTP.

# theme

  • Type: Plugin
  • Default: the default theme

Specify a theme that should be used. A theme is simply a regular miniPress plugin. You specify a theme like this:

module.exports = {
  theme: ['@minipress/theme-default', { /* options*/ }]
}

options are optional

If you want to learn more you should head over to the Theme Guide.

# configureSiteData

  • Type: SiteDataFn
  • Default: no additional configuration is performed

Modify the site data. The site data will be available via $minipress.site and exposes meta data for the whole site.

type SiteDataFn = (site: object) => Promise<object | void>

configureSiteData must be a function which takes the current site data object. You can directly mutate site or return a completely new object.

Example:

module.exports = {
  configureSiteData(site) {
    site.productName = 'miniPress'
  }
}

Now you can access the value for productName in your *.vue/*.md-files by using $minipress.site.productName.

# transformPage

  • Type: (page: Page) => Promise<void>
  • Default: no transformation is applied sss Allows you to transform a page. transformPage must be an async-function which gets a Page. You can modify the page within this function â€“ for example you can modify the frontmatter object in order to delete or add properties to it.

Example:

const fs = require('fs-extra')
module.exports = {
  async transformPage(page) {
    if(page._filePath != null) {
      const stats = await fs.stat(page._filePath)
      page.properties.lastModified = stats.mtime.toISOString()
    }
  }
}

This transformPage-function is setting properties.lastModified to the date when the page was last modified. Setting lastModified on properties will make lastModified available as:

{{ $minipress.page.lastModified }}

# plugins

  • Type: Plugin[]
  • Default: []

Specifies plugins that should be applied. Plugins are distributed as NPM-packages. This is how you specify plugins in your config.js file:

module.exports = {
  plugins: [
    ['@minipress/plugin-additional-pages', { /* options */ }],
    // more plugins
  ]
}

You can write your own plugins. There is a dedicated Plugin Guide available.

# head

Customize the value of the <title>-tag and the content of <meta name="Description">.

  • Type: HeadConfig
  • Default:
{
  title: 'miniPress',
  description: 'My miniPress Website'
}

The shape of HeadConfig looks like this:

interface HeadConfig {
  title?: string
  description?: string
}

# dynamicModules

  • Type: ({ context: Minipress }) => DynamicModules
  • Default: () => {}

Specify custom dynamic modules.

type DynamicModules = {
  [name: string]: string
}

dynamicModules is a function that will be called with the current miniPress-instance. You have to return an object mapping module name to the code of a module that should be injected in the app.

Example:

const package = require('./package.json')

module.exports = {
  dynamicModules($site) {
    return {
      package: `export default ${JSON.stringify(package)};`
    }
  }
}

This creates a module called package which exports the contents of your package.json. In your *.vue/*.js-files you can use this module like this:

import package from '#minipress/dynamic/package'
console.log('The contents of package.json:', package)

Dynamic modules are created at build time.

# components/

Any *.vue file placed in the components directory will be registered globally. You can create more directories inside the components-directory. Global components can be used in your pages and in other *.vue-files. The sample directory structure shown above contains three component files: table/index.vue, table/row.vue and my-button.vue. miniPress will register these components under the following name:

  • my-button.vue will be named MyButton
  • table/index.vue will be named Table-Index
  • table/row.vue will be named Table-Row

# layouts/

Any *vue file placed in this directory are registered as layout components. The example layouts-directory above contains two layouts. When a page is rendered the default-layout will be used. Use frontmatter to override this:

---
layout: fullscreen
